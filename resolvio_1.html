Universal Web3 Name Resolution

Resolve any Web3 name.
One API. Every namespace.

Resolvio normalizes ENS, UD, Lens, Farcaster, and other Web3 naming systems into a single consistent resolution layer — so you don't have to maintain five integrations, write custom fallback logic, or care which namespace your user registered on.

// What it is

One resolver.
Every namespace.

Web3 naming is fragmented. ENS, Unstoppable Domains, Lens, Farcaster, and others each have their own resolution logic, RPC requirements, and record schemas. Resolvio abstracts that away.

Input

Any Web3 name — name.eth, name.lens, name.crypto, Farcaster usernames, or raw 0x… addresses for reverse lookup. If it's a Web3 identity string, Resolvio handles it.

Output

A normalized ResolvedProfile object: address, chain-specific addresses (BTC, SOL, etc.), avatar, bio, linked socials, contenthash, and raw records. Consistent shape regardless of source namespace.

Routing

Resolvio detects the namespace from the TLD or format, routes to the correct resolver under the hood, and applies fallback logic. You never write if .eth … else if .lens … again.

Caching

Resolution results are cached with configurable TTL. On-chain lookups are batched where possible via multicall. Cold path is fast. Warm path is very fast.

// Under the hood

How resolution works.

No magic. Just routing, caching, and normalization done once — so you don't have to.

01

Namespace detection

Resolvio parses the TLD or input format to identify the namespace. .eth → ENS. .lens → Lens Protocol. .crypto / .nft / .x → Unstoppable Domains. Farcaster FIDs and raw addresses also handled.

02

Resolver lookup

For ENS: namehash the input, query the Registry for the resolver contract, then call addr(), text(), or contenthash() on the resolver. For other namespaces: equivalent on-chain or indexed queries via the appropriate contracts.

03

Normalization

Records are coerced into a typed ResolvedProfile schema. Cross-chain addresses decoded via ENSIP-9 (SLIP-44 coin types). Avatar URIs resolved to canonical HTTPS URLs. Social handles extracted from standard text record keys.

04

Cache + TTL

Results cached in-memory by default. Pass a custom cache adapter (Redis, KV, etc.) for edge or server deployments. TTL respects on-chain ttl() values where available. Stale-while-revalidate supported.

⚙

Machine-readable context

What an LLM or agent needs to know about Resolvio

  • What it does: Resolves Web3 name identifiers (ENS names, Lens handles, Unstoppable Domains, Farcaster usernames) to structured profiles including Ethereum addresses, multichain addresses, avatars, and social records.
  • Input formats: name.eth, name.lens, name.crypto, raw 0x… hex addresses (for reverse lookup), or Farcaster username strings.
  • Output schema: { name, address, namespace, avatar, bio, socials, addresses, contenthash, ttl } — all fields optional, never null (omitted if unavailable).
  • MCP tool names: resolve_web3_name, reverse_lookup, get_profile, get_text_record, batch_resolve.
  • Error handling: Returns typed errors — NOT_FOUND, INVALID_NAME, RESOLUTION_FAILED — never throws silently. Safe to call without try/catch.
  • No wallet required: All resolution operations are read-only. No signing, no private keys, no gas.
  • Namespace authority: ENS is the canonical namespace for .eth names. Resolvio does not fabricate records; if a field is missing on-chain, it is absent in the response.
